01 — What it is

One account, every business you run.

Kasero is for food vendors, artisans, and small retailers — the operator behind the counter who needs to ring up a sale in two taps, track stock, and see the day's numbers without a back office.

It started at La Feria de Barranco, a weekend artisan fair in Lima — vendors running stalls on a paper notebook, a phone calculator, and Yape for payments. That stall is the user every screen is measured against.

Multi-business

Run several businesses from one login, each with its own catalog, team, currency, and locale.

Product catalog

AI-generated product icons, categories, stock levels, and barcode scan + generate.

Inventory

Track stock and supplier orders; the register decrements stock atomically on every sale.

Sales register

Open and close sessions, ring up sales, settle cash with change due, and roll up daily totals.

Expenses

Log costs against categories and see the month's spend at a glance, in the business's own currency.

Team & ownership

Invite partners and employees with role-based access — and hand over the whole business with an atomic ownership transfer.

Installable as a PWA, it keeps working when the venue Wi-Fi drops — the thing a market stall actually needs at the worst possible moment.

02 — The shape of the system

One repo, two apps, one shared spine.

An npm-workspaces monorepo. A single source of truth — the database schema, types, validation, the message-code vocabulary, and the locale registry — lives in packages/shared and is consumed by both apps through TypeScript project references. Define a shape once; the client and the server can't disagree about it.

The single-origin trick. At build time a script folds the built SPA into the API's public/ folder, so one deployment serves both /api/* and the app shell from the same origin.
Decision

Serve the SPA and the API from one origin.

Why

Two origins means CORS, third-party-cookie breakage, and two deploys to keep in sync.

Impact

Cookies "just work," no CORS config, one thing to deploy and roll back.

apps/api/src/app/api/businesses/[businessId]/categories/route.ts
const createCategorySchema = z.object({
  name: Schemas.name().max(50),
})

export const POST = withBusinessAuth(async (request, access) => {
  if (!canManageBusiness(access.role)) {
    return errorResponse(ApiMessageCode.FORBIDDEN, 403)
  }

  const validation = createCategorySchema.safeParse(await request.json())
  if (!validation.success) {
    return validationError(validation)   // Zod issues -> ApiMessageCode, auto-translated
  }

  // ... insert, then publish a realtime hint ...
  return successResponse({ category: newCategory })
})
Decision

Every route returns an ApiMessageCode envelope, never a raw English string.

Why

An error message hardcoded on the server can never be shown in the user's language.

Impact

One code crosses the wire; the client renders it in any of 11 languages. Validation comes free from the Zod schema.

04 — Production-grade engineering

Why it's built the way it is.

Five decisions, each one a constraint that forced a choice — and what that choice buys a real operator standing at a stall.

4.1

One string, eleven languages

User-facing text never ships as a literal — not in the UI, not in an API response. Strings go through formatMessage; the server speaks message codes. A single registry drives all 11 locales.

apps/web/src/hooks/useUpdateBusiness.ts
} catch (err) {
  console.error('Update business error:', err)
  setError(
    err instanceof ApiError && err.envelope
      ? translateApiMessage(err.envelope)   // server code -> this user's language
      : '',
  )
  return false
}
Decision

Translate server errors on the client via useApiMessage; add a language by copying one JSON file and editing the registry.

Why

Kasero's operators aren't all English speakers — they're in Lima, Manila, Kyoto. Language is a property of the person.

Impact

de · en · es · fil · fr · it · ja · ko · pt · vi · zh, all from one source. No string is ever stranded in English.

4.2

Money is a property of the business, not the person

Formatting — currency, dates, times — is keyed off the business's locale and currency, resolved once by a hook and never hardcoded in a component.

apps/web/src/components/expenses/ExpenseTotalsStrip.tsx
const { formatCurrency } = useBusinessFormat()
if (loading || summary === null) return null

return (
  <div className="expenses-totals-strip">
    {t.formatMessage(
      { id: 'expenses.totals_strip_month' },
      { amount: formatCurrency(summary.totalExpenses) }  // $9.99  or  S/ 9.99
    )}
  </div>
)
Decision

Separate axes: users.language (a person) vs businesses.locale/currency (a business).

Why

A Peruvian stall must show S/ to its English-speaking accountant and its Spanish-speaking cashier alike.

Impact

Correct money everywhere with zero per-component locale logic — never a $ baked into a Peruvian receipt.

4.3

Await what can fail; fire-and-forget what can't

A sale's commit is awaited so an error can surface before the receipt — but the stock-cache refresh that follows is non-blocking. The user moves to the success step immediately; the picker catches up on its own.

apps/web/src/components/sales/cart-modal/ChargeButton.tsx

Decision

Await the mutation; void the follow-up cache refresh.

Why

A failed charge must be caught and shown; a slow cache refresh must never make the next customer wait.

Impact

The line keeps moving. Correctness where it matters, speed where it doesn't.

4.4

Realtime that fails the right way

Realtime events ride SSE over Redis. But not every event is equal: a "list changed" nudge is cosmetic, while a revoked session after an ownership handover is a security boundary. So the publisher has two doors.

apps/api/src/app/api/transfer/accept/route.ts
// Critical publishes -- fail-closed. If either throws, return 503 so
// the caller knows the ownership event was not delivered.
try {
  await Promise.all([
    publishCriticalToUser(newOwnerId,   { type: 'ownership.transferred', businessId, role: 'new_owner' }, originDeviceId),
    publishCriticalToUser(formerOwnerId, { type: 'session.revoked', businessId, reason: 'ownership_transferred' }, originDeviceId),
  ])
} catch (err) {
  if (err instanceof RealtimeUnavailableError) {
    return errorResponse(ApiMessageCode.REALTIME_PUBLISH_UNAVAILABLE, 503)
  }
  throw err
}
// Non-critical publishes -- fail-open.
await publishToUser(newOwnerId, { type: 'business.list.changed', reason: 'added' }, originDeviceId)
Decision

Two publish paths: publishToUser (fail-open) and publishCriticalToUser (fail-closed → 503).

Why

A dropped UI hint is invisible; a dropped session.revoked leaves a stale session live on someone else's device.

Impact

Cosmetic events never break a request; security events refuse to silently disappear — the caller is told to retry.

4.5

The unglamorous decisions that read as "production"

The kind of choices a senior reviewer looks for — each one a small refusal to take the easy, fragile path.

Fail-closed rate limiting

If the limiter backend is unreachable, return 503 instead of quietly serving traffic with brute-force protection switched off.

Same-origin guard, before auth

enforceSameOrigin checks Origin/Referer against the forwarded host before the auth lookup, so a cross-site request can't even probe timing.

Atomic ownership transfer

In-transaction .returning() row checks refuse to create a double-owner or hand a business to a just-disabled account — illegal states made unrepresentable.

Apple secret mints itself

Apple's OAuth client secret is a JWT that expires in ≤6 months; it's regenerated at module load so no human is ever paged to rotate it.

Passwordless by design — the password column was dropped in a migration. Auth is better-auth: email-OTP plus Google and Apple, on database-backed sessions.

05 — Knowing who it's for

The hardest thing we built was the judgment to delete.

"Small business" is a wide net. Over a few months the codebase quietly accreted what a different user needed — and the most production-grade decision in the project was to cut it back out.

The drift

A named-supplier book, a purchase-order wizard, packaged-goods barcode printing — roughly 13,000 lines that re-shaped an artisan-fair POS into a convenience-store back office. Nobody decided to build a different product; the features just arrived, one reasonable-seeming step at a time.

A candid internal review named it: a codebase that doesn't know who its user is will accrete features that constrain a different user than the one you started with.

The cut

So the supplier and purchase-order machinery was removed and the product refocused on the in-person vendor it was built for. The parts that earned their place stayed and deepened — offline-first, the 11-locale i18n, the atomic sale and ownership-transfer correctness.

Why it matters

The discipline to delete working code that serves the wrong user is the same discipline that ships production-grade software. Knowing what not to build is half the craft.

07 — In closing

How it's built is the point.

A market stall doesn't care about your architecture — until the Wi-Fi drops, the currency is wrong, or a session that should be dead is still alive. Kasero is engineered for those moments: offline-first, locale-correct, fail-closed where it counts.

Distributed teams, transparent process, production-grade software — the same standard, applied to a product real people run their livelihoods on.
